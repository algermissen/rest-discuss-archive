Well, now that's a completely clueless statement.  The point of the
definitions is that the interface does not define the exact meaning,
leaving meaning to be defined by the nature of the links between
the information.  A server does the interpolation between the contents
of a PUT and the model hidden behind the server -- saying that isn't
defined by REST is just goofy.  I have no problems building PUT,
containers, or any of the other things that you claim are not defined
by REST, since those are implementation details that are defined within
HTTP by the origin server and not by the client.  I don't even need to
use webdav, though it has its definitions as well.  The only problems
I run into are buggy browsers that never managed to implement basic
HTTP correctly, let alone the advanced features.

If you don't like the generic interface, then just implement a
distributed file system.  That has all the features you are looking
for in a concrete model.
