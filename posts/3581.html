>> Theoretically, you simply PUT something that needs containment,
>> as evidenced by the slash in the path.  However, most webdav
>> implementers believed that a collection should exist prior to
>> a PUT within it, since that makes it easier to catch user mistakes
>> such as mistyping a long URI.  Hence, MKCOL.
>>
>> REST will happily accept either alternative, providing that the
>> method is standardized (has the same meaning) for all resources
>> (even if some may not allow it).
>
> But does HTTP happily accept either alternative?  Or does HTTP
> standarize on one or the other method?

HTTP doesn't tell applications how to do a sequence of actions.
It only defines what each action means and how the client should
interpret the response to each action.  WebDAV is a little more
explicit about sequences, which is why it prefers MKCOL and would
probably reject a deep PUT.

>> If you delete the container, it may have adverse effects on other
>> links.  For example, if the server automatically maintains links,
>> then you should not delete the container unless you want all of
>> those links to go away.  So, delete each resource individually.
>
> What do you mean by "the server automatically maintains links?"

Many content management products (like Day Software's Communique)
include an automatic link maintenance function that will recognize
a delete on a resource and either note all links to it as being
broken (and thus scheduled for review) or simply deactivate those
links so that they don't show up on the published pages until someone
either changes the destination or restores the deleted resource.
