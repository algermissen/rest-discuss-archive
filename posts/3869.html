> The genius of ReST is its recognition of the value of a standardized
> resource interface in a heterogeneous large scale networked 
> environments
> and its attempt to articulate such an interface.  Its tragic flaw is 
> that
> it did not standardize enough of the *semantics* of this interface to
> achieve the value it promised.  To be perfectly clear about the 
> problem:
>
> * Without an *explicit* model of exactly how the server maps the state 
> of a
> resource into a response to a GET request, a client can have 
> ABSOLUTELY NO
> IDEA what the content of that response means; it is simply has bits 
> that
> can mean anything at all.

What does "mean" mean in this context?

> * Without an *explicit* model of exactly how the server uses 
> PUT/POST-ed
> content to change the state of a resource, a client can have 
> ABSOLUTELY NO
> IDEA what the effect of a PUT or POST actually will be.

That just says that REST doesn't mandate one single model.  Nothing
prevents the server from establishing a model and telling the client
what it is.  That is how hypertext works: the engine of application
state change.

> Roy thinks these explicit models should be entirely arbitrary and 
> specified
> at the whim of the server operator on a per-URI basis.  I think these
> models should be standardized in some manner and be URI independent.

They aren't entirely arbitrary; they are merely chosen by the server.
In some cases they will be selected from a standard (such as webdav).
In other cases they will be defined by an interface language (like 
WSDL).

> Because Roy's resources are so abstract, access to them has to be 
> mediated
> by agents of arbitrary power and complexity.

Ah, so httpd modules must be gods.

> Allowing clients to interact
> with agents of this power and complexity through HTTP then requires 
> that
> service operators be permitted to overload HTTP methods to have 
> essentially
> arbitrary meanings.

That doesn't follow at all.  Service operators control the 
implementation
of HTTP methods.  Methods get overloaded because service operators do 
want
additional semantics, but normal people don't know how to extend them 
and
implementers can't read protocol specs.  It is an artifact of the social
process surrounding standardized protocols.  You can't fix it by
forbidding the extension of methods.

> But resources don't really need to be so abstract and the meanings of 
> HTTP
> methods don't need to be so loose.  Programs don't actually manipulate
> "concepts" or "non-virtual objects;" they just manipulate bits.

Right.  It's people that manipulate concepts and non-virtual objects.

>   It would
> be more realistic to say that an HTTP resource is simply a set of bits
> whose syntactic structure is enforced by an HTTP server and whose 
> meaning
> is determined entirely by agents which interact with this server.

No it wouldn't, since such a definition ignores the ability of people
to anticipate changes that have not yet occurred.  Time is essential
to the meaning of a hypertext link, and thus the definition of resource.
HTTP servers store representations, not resources.

>   To say
> otherwise is to say that there is a meaningful difference in effect 
> between
> having an agent poll a resource and act when it changes and having a 
> server
> notify an agent to act whenever a resource is changed or to say that 
> there
> is a meaningful difference between such an agent residing on the HTTP
> server itself and such an agent residing on another machine across the
> Internet and interacting via HTTP.

Or simply that resource isn't what you think it means.

> Viewed this way, it makes no sense to talk about some single *special*
> hidden representation to which a single special HTTP server agent 
> mediates
> access.  Instead we should recognize that agents on the server and 
> across
> the Internet can use any data structures they want for their 
> activities but
> that their doing so is really beside the point.  The function of an 
> HTTP
> server as such is simply to provide a shared blackboard for

The blackboard model is another architectural style with a lot of
interesting work associated with it.  Tuple-spaces use that.  The
problem with blackboards isn't just a lack of a data model;
they also have centralized identification and lack metadata.
